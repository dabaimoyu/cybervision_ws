# 帧率配置与修改问题总结

## 📋 概述

**时间**: 2024 年 3 月  
**影响模块**: `cybervision_infer_node`、`cybervision_render_node`  
**关键指标**: NFR-2 - 系统吞吐量 ≥ 60 FPS  

---

## 🎯 帧率目标与要求

### 核心性能指标（NFR）

根据项目需求定义：

```cpp
// performance_monitor.hpp 中的定义
static constexpr double TARGET_MIN_FPS = 60.0;              // 最低 60 FPS
static constexpr double TARGET_MAX_INFERENCE_TIME_MS = 16.67; // 最大推理耗时 (1/60 秒)
static constexpr double TARGET_MAX_LATENCY_MS = 60.0;      // 最大端到端延迟
```

### 帧率层级

系统涉及三个层级的帧率：

1. **视频源帧率**: 输入视频的原始 FPS（约 30 FPS）
2. **推理节点帧率**: TensorRT 推理速度（目标 ≥ 60 FPS）
3. **渲染节点帧率**: OSG + Qt 渲染刷新率（目标 60 FPS）

## 🔧 帧率配置位置

### 1. 推理节点 (Infer Node)

**文件**: `src/infer_node/main.cpp`

**控制逻辑**:
```cpp
// 没有显式的 FPS 限制器
// 帧率由以下因素决定：
// 1. 视频源的 PTS（Presentation Time Stamp）
// 2. VideoStreamer 的帧率控制
// 3. TensorRT 推理速度
```

**实际表现**:
- 视频源 FPS: ~30 FPS（由视频文件决定）
- TensorRT 推理速度：~180-200 FPS（空载测试）
- 实际处理帧率：~18-20 FPS（受限于视频源）

### 2. 渲染节点 (Render Node)

**文件**: `src/render_node/main.cpp` (第 109 行)

**配置代码**:
```cpp
// 用 QTimer 驱动主循环，让 Qt 事件循环负责调度（每 16ms 约 60 FPS）
QTimer* timer = new QTimer();
QObject::connect(timer, &QTimer::timeout, [this, &loop_count]() {
    // ... 渲染逻辑 ...
});
timer->start(16);  // 16ms ≈ 60 FPS
```

**设计原理**:
- Qt 定时器周期：16ms
- 理论刷新率：1000ms / 16ms ≈ 62.5 FPS
- 实际刷新率：约 60 FPS（Qt 事件循环开销）

## 🐛 帧率相关问题

### 问题 1: 推理节点帧率不达标

#### 现象
```log
实时 FPS:   18.426
⚠️  警告：部分性能指标未达标!
  ❌ FPS: 18.426 < 60
```

#### 根本原因
1. **视频源限制**: 输入视频本身只有 ~30 FPS
2. **PTS 同步机制**: VideoStreamer 严格按照视频 PTS 输出帧
3. **无法跳帧**: 当前实现逐帧处理，不会跳过任何帧

#### VideoStreamer 帧率控制
```cpp
// video_streamer.cpp
bool VideoStreamer::readFrame(AVFrame*& frame, int64_t& pts) {
    // ...
    // 根据 PTS 计算等待时间
    double wait_time = pts_time - clock_time;
    if (wait_time > 0) {
        std::this_thread::sleep_for(std::chrono::microseconds(
            static_cast<int>(wait_time * 1e6)
        ));
    }
}
```

**结果**: 系统帧率被视频源锁定在 ~30 FPS

### 问题 2: 渲染节点与实际数据不同步

#### 现象
- Qt 定时器：固定 60 FPS（16ms）
- 感知数据：约 18-20 FPS（实际视频帧率）
- OSG 渲染：每帧都调用，但数据可能不变

#### 潜在问题
```cpp
// main.cpp - QTimer 回调
if (frame && !frame->objects.empty()) {
    osg_renderer_->updateObjects(frame->objects, frame->frame_id);
    osg_renderer_->renderFrame();  // 即使数据没变也会渲染
}
```

**浪费**: 当没有新数据时，仍在渲染旧数据

## 🛠️ 帧率修改方案

### 方案 A: 提高推理节点帧率（推荐用于测试）

#### 方法 1: 禁用 PTS 同步（仅用于性能测试）

**修改文件**: `src/infer_node/video_streamer.cpp`

```cpp
// ❌ 不推荐：完全移除 PTS 控制
bool VideoStreamer::readFrame(AVFrame*& frame, int64_t& pts) {
    // ... 读取帧 ...
    
    // 注释掉 PTS 同步
    // double wait_time = pts_time - clock_time;
    // if (wait_time > 0) {
    //     std::this_thread::sleep_for(...);
    // }
    
    return true;  // 立即返回下一帧
}
```

**效果**: 
- ✅ 推理速度可达 180-200 FPS
- ❌ 不符合实际应用场景（真实摄像头是实时流）

#### 方法 2: 使用高速视频源

**推荐做法**: 使用 60 FPS 或更高帧率的视频进行测试

```bash
# 检查视频 FPS
ffprobe -v error -select_streams v:0 -show_entries stream=r_frame_rate \
  -of default=noprint_wrappers=1:nokey=1 input.mp4
```

**优势**:
- ✅ 符合实际应用场景
- ✅ 可以验证系统是否达到 60 FPS 设计目标

### 方案 B: 优化渲染节点帧率匹配

#### 方法 1: 动态调整渲染频率

**修改文件**: `src/render_node/main.cpp`

```cpp
// 记录上次渲染的帧 ID
int64_t last_rendered_frame_id = -1;

QObject::connect(timer, &QTimer::timeout, [this, &loop_count, &last_rendered_frame_id]() {
    auto frame = double_buffer_->read();
    
    // 只在有新数据时才渲染
    if (frame && frame->frame_id != last_rendered_frame_id) {
        osg_renderer_->updateObjects(frame->objects, frame->frame_id);
        osg_renderer_->renderFrame();
        last_rendered_frame_id = frame->frame_id;
    }
    
    swapBuffers(*double_buffer_);
});
timer->start(16);  // 保持 60 FPS 定时器
```

**优势**:
- ✅ 避免无效渲染
- ✅ 降低 GPU 负载
- ✅ 节省 CPU 资源

#### 方法 2: 根据数据帧率自适应渲染

```cpp
// 计算最近 10 帧的平均数据到达间隔
std::vector<double> frame_intervals;
int64_t last_frame_id = -1;
rclcpp::Time last_frame_time;

void perceptionCallback(const PerceptionResult::SharedPtr msg) {
    rclcpp::Time now = this->now();
    if (last_frame_id >= 0) {
        double interval = (now - last_frame_time).seconds() * 1000.0;  // ms
        frame_intervals.push_back(interval);
        if (frame_intervals.size() > 10) {
            frame_intervals.erase(frame_intervals.begin());
        }
        
        // 计算平均间隔并调整渲染定时器
        if (!frame_intervals.empty()) {
            double avg_interval = std::accumulate(frame_intervals.begin(), 
                                                  frame_intervals.end(), 0.0) / frame_intervals.size();
            int timer_period = std::min(16, static_cast<int>(avg_interval));
            timer->start(timer_period);
        }
    }
    last_frame_id = msg->frame_id;
    last_frame_time = now;
    // ...
}
```

**优势**:
- ✅ 自动匹配数据源帧率
- ✅ 减少不必要的渲染
- ✅ 降低系统功耗

### 方案 C: 固定 60 FPS 渲染（当前默认）

**配置**:
```cpp
// src/render_node/main.cpp
timer->start(16);  // 固定 60 FPS
```

**适用场景**:
- ✅ 演示和展示（流畅的视觉效果）
- ✅ 需要高刷新率的交互场景
- ❌ 数据处理场景（可能造成资源浪费）

| 方案 | 优点 | 缺点 | 适用场景 |

## 📝 常见问题 FAQ

### Q1: 为什么推理节点 FPS 只有 18-20？

**A**: 这是正常的！原因：
1. 视频源本身约 30 FPS
2. VideoStreamer 严格按 PTS 同步
3. 系统不会跳帧处理

**解决方案**: 如需测试 60 FPS 能力，请使用 60 FPS 视频源。

### Q2: 能否让推理速度超过视频源帧率？

**A**: 理论上可以，但不推荐：
- ❌ 跳帧会丢失检测信息
- ❌ 不符合自动驾驶安全要求
- ✅ 正确做法：使用更高帧率的相机

### Q3: 渲染节点为什么要固定 60 FPS？

**A**: 设计考虑：
1. **视觉流畅性**: 60 FPS 是人眼感知的流畅标准
2. **交互响应**: 即使用户操作（旋转、缩放）也需要高刷新率
3. **解耦设计**: 渲染帧率独立于数据帧率

### Q4: 如何验证系统真正达到 60 FPS？

**A**: 三种方法：
1. **日志查看**: `实时 FPS: XX.XXX`
2. **OSG 统计**: 按 's' 键显示 FPS
3. **外部工具**: `ros2 topic hz /perception_results`

## 🎓 经验教训

### 1. 区分"处理能力"与"实际帧率"

**教训**: 
- TensorRT 推理能力：200 FPS（处理能力）
- 视频源帧率：30 FPS（实际输入）
- 系统帧率：30 FPS（受限于最慢环节）

**启示**: 性能测试必须明确测试目标
- 基准测试：测最大处理能力（禁用 PTS）
- 系统测试：测实际工作帧率（启用 PTS）

### 2. 帧率匹配的重要性

**问题**: 渲染 60 FPS vs 数据 20 FPS = 67% 的渲染是无效的

**优化方向**:
- 只在有新数据时渲染
- 或使用插值平滑（游戏常用技术）

### 3. PTS 同步的必要性

**原因**: 
- ✅ 保证时间一致性（对自动驾驶至关重要）
- ✅ 模拟真实传感器行为
- ❌ 限制了测试时的最高帧率

**权衡**: 
- 开发阶段：可暂时禁用以测试极限性能
- 生产环境：必须启用以确保安全
